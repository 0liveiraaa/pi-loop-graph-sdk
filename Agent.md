## 一、项目是什么

Loop Graph SDK 是一个 **agent 编排框架**，基于 pi extension 机制构建。

它的本质是**对 Skill 概念的深化**——将 Skill 从静态 Markdown 指令落实为可被软件工程管理的代码实体：一张可运行的、可被 LLM 像调用普通 skill 一样调用的**多源有向回路图**。

**它是**：编排层。图表征一组业务逻辑但本身不执行业务逻辑——只承载节点、边、路由器，为 agent 实例提供运行空间。决定"什么时候跑什么"，不干涉节点内部的 LLM 推理。

**它不是**：推理层、业务层、全局运行时。每个业务 extension 通过 `createLoopGraphExtension(pi)` 创建自己的隔离实例。SDK 不与任何特定领域（如复习、客服）绑定——pi-review-agent 仅为示范 SDK 通用性的用例。

## 二、核心心智模型

### 回路图 = 暴露 workflow 状态转移，隐藏 ReAct 循环

相较于 DAG，图**支持回路**。这背后是两层循环粒度的严格划分：

| 粒度                                         | 归属                     | 原因                                                                             |
| -------------------------------------------- | ------------------------ | -------------------------------------------------------------------------------- |
| 单步推理循环（ReAct）                        | **内化在节点内部** | 它是"怎么做"——不应污染图结构                                                   |
| 跨阶段、多子目标的迭代（如生成→审查→修改） | **表达为图边回路** | 它是"做什么、何时做"——必须在图拓扑中显式，保证每个阶段独立职责、可观测、可维护 |

框架在 ReAct 之上抽象：编排不推理。

### 帧栈 = 信息流动的调用栈

AgentInstance 持有一个有序帧栈，模仿内存调用栈的工作方式。每进入一个节点栈上生长一层，离开时由边折叠栈顶层。历史留档可追溯，不可悄悄篡改。

设计在三重力量之间找平衡：

- **KV cache 需要连续生长** → 帧栈持续追加
- **上下文应当精简** → 旧 ReAct 折叠为摘要帧；子图隔离栈整体归约
- **信息应当充足** → 折叠后的摘要帧保留关键结果，不可丢失

子图隔离栈（父图 frames 对子图不可见，子图结束后整体归约）是同一哲学的延伸：**子图是函数式调用边界**——父图只看调用结果，不偷看子图内部历史。顶层图调用 = 没有调用者的子图调用。

### 模块隔离与职责分配

- **Edge.guard 只依赖 NodeCompletion**，不触碰 AgentInstance 内部状态——模仿模块间接口隔离
- **Node 自声明一切**（tools、subGoal、skill），Edge 不负责准备——「谁拥有信息，谁就声明信息」
- **Node 只产出原始 NodeCompletion**，Edge.migrate 负责折叠进帧栈——「边是完整决策」：同一次完成，走不同边可以有不同的"记忆"
- **入口使用 Entry**（处理 trigger/background），不复用 Edge（处理 NodeCompletion）——「同构不是目的，类型自洽优先」

## 三、不可妥协的原则

1. **编排不推理**。框架不调用 LLM 做摘要、评估或决策。帧栈的 summary 是纯字符串。路由裁决是代码函数。
2. **无隐式全局状态**。所有跨节点状态必须显式存在于帧栈中。没有隐藏在闭包、模块变量或副作用中的数据传递。
3. **图即真相**。所有跨阶段状态迁移必须表达为图的边和路由。不允许节点内部隐藏跨节点迁移逻辑。
4. **隔离即边界**。子图创建新 AgentInstance，`frames = []`。父图不可窥探子图内部。
5. **约束架构骨架，开放业务实现**。框架通过类型固化核心契约（Node、Edge、Router 的接口），但不阻拦合理的定制化需求。所有扩展点都是纯函数。
6. **追加不注入**。SDK 永远不修改 system prompt。所有上下文操作只在消息流侧进行（如 projection 钩子追加 COMPLETED/CURRENT 段）。追加大于注入（append > inject）,这是栈思想的体现,要牢记栈心智模型,每次涉及到上下文管理都要回顾。

## 五、文档纪律

1. **代码变行为，文档必须变。** 任何改变行为、API 或已验证特性的代码变更，必须同步更新 `docs/形态/implementation-status.md`。非协商。
2. **决策必须落笔。** 讨论中达成的设计结论，不得仅存于聊天记录或 issue。追加到 `docs/设计/loop-graph-sdk-design.md` 附录的"设计演进"部分，或创建独立设计文档。
3. **文档必须能被找到。** 新文档必须在 `docs/README.md` 索引中注册。没有入口链接的文档等于不存在。
4. **废弃文档必须归档。** 不留在原目录误导后来者。

## 六、当前状态

- **版本**：v0.1.0，单 agent MVP
- **已验证**：帧栈折叠、子图隔离、哨兵投影、完成度验证、工厂实例隔离、包边界分离
- **已知缺口**（详见 `docs/形态/implementation-status.md` 第五节）：agent-choice 路由、callTool、多 skill、schema helper、session 续跑、defaultTools 不流入 skill 节点
- **能力债优先级**：先保障单 agent 编排完整性（callTool → 多 skill → agent-choice → schema helper）。多 agent 通讯（`docs/设计/communication-design.md`）为核心远期愿景，单 agent MVP 是第一步。
- **质量承诺**：可靠性优先。已知问题必须修复且必须在文档中声明，才可声称 stable。

---

## 七、使用这份宪法

给接手项目的 AI agent：

1. 读完本文——获得心智模型和边界。
2. 根据任务，按第四节地图定位文档。
3. 做任何改动前，检查第五节——有没有文档需要同步更新。
4. 做设计决策时，对照第三节——方案是否违反不可妥协原则。
5. 需修订宪法时，在修改建议中标注，由维护者决定。
