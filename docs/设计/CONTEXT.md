# 工作流编排(pi loop graph sdk)

## 回路图 (Loop Graph)

回路图是对 Skill 概念的深化——将一个 Skill 从静态 Markdown 指令落实为可被软件工程管理的代码实体。它本质上是一个可运行的**多源有向图**，可以被 LLM 像调用普通 skill 一样调用。调用后，agent 实例进入图，按流水线一步步执行直到任务完成、返回结果。


回路图表征了一组业务逻辑但本身不执行业务逻辑，只承载节点、边、路由器、共享环境，并为 agent 实例提供运行空间。

**要点**:
- 图应支持循环
    相较于有向无环图（DAG），允许边构成回路更贴合设计哲学:暴露workflow下的工作状态转移,隐藏reAct下的loop，且能让跨阶段的迭代（如生成‑审查‑修改）直接在图拓扑中表达，避免将业务逻辑沉入不可见的节点内部。

- 循环粒度严格划分
    - 单步推理循环（ReAct） 必须内化在 Agent 节点 内部，作为节点完成单一子目标的执行方式，不污染图结构。
    - 跨阶段的、多子目标的迭代循环 必须以 图边回路 表达，保证每个阶段的独立职责、可观测性与可维护性。

- 图即模块，可被复用
    一组有明确边界的回路图（如“执行‑检查”循环）可以被封装为 图节点（复合节点） ，对外呈现普通节点的接口，从而构建可组合、可积累的技能库，实现技能工程化管理。

- **多源头**：每个源头代表一个独立的 agent 实例。
- **状态迁移**：agent 实例在图中的流动由路由器根据边发布的迁移请求驱动。
- **共享环境**：可包含跨 agent 的通信设施（见"通讯"）。
- **入口**：调用即入口。无需独立的 Entry 实体——Graph 自身通过入口配置匹配调用来源并创建 agent 实例。
- **终点**：END 是一个 sentinel 标记（非虚拟节点），用于编译器区分"合法终止"和"遗漏边"。

*Avoid: Loop 工程, harness, 工作流链*

回路图和技能回路是同义词。

## Agent 实例 (Agent Instance)

Agent 实例是回路图中的**活动主体**。它承载跨节点的连续状态,并在图中从一个节点迁移到另一个节点。
每个 agent 实例拥有独立的身份和全局意图,通过与节点提供的局部约束组合,形成当前阶段的完整运行上下文。

**基本要素**:

1. **目标**:回路级的总体目标,是 agent 在所有节点中的不变追求。
2. **上下文 (Context)**:agent 实例的累积记忆,采用**上下文层 (Context Layer)** 模型管理。每进入一个节点,上下文中新增一个隔离层;节点内的 LLM 调用和工具交互都在该层发生。节点完成时,该层的中间过程(ReAct 轨迹、临时推理)被折叠,只保留产出摘要挂到上层。Edge 的迁移规则只需指定产出摘要应包含的字段,无需声明丢弃哪些中间状态--层隔离天然处理清理。上层的全局记忆(profile、session、学习画像)持续累积,不受当前节点折叠影响。
3. **LLM 执行 (llm_execute)**:绑定在 agent 实例上的 LLM 对话能力。节点通过组合规则限制其调用范围(工具白名单、技能注入),但调用能力本身属于 agent 实例。一次 `run` 对应一次完整的 LLM 对话(多轮工具调用直到停止)。节点内部可以反复调用它来实现循环。
4. **可注册工具**:全局可用的基础工具(如记忆读写、向用户提问)。
5. **机制**:全局横切面基础设施,满足条件后自动执行(如上下文压缩、反思、目标重规划)。

- agent 实例不直接拥有阶段性的工作状态,这些状态由节点管理。
- 在多 agent 场景中,每个实例独立运行,但可通过通讯设施进行交流协作

*Avoid: 全局单例, 超级 prompt, 会话对象*

## 节点 (Node)

节点是回路图中的**可运行工作阶段**。它定义了 agent 实例在该阶段需要完成的事,并管理 agent 实例在此阶段的局部状态。
节点本身是阶段模板;当 agent 实例进入时,节点会为其创建一个隔离的**节点会话**,承载该实例在此节点内的 ReAct 状态、临时数据和子目标进展。

**节点声明**:

- **子目标**:该阶段需要达成的具体目标,与 agent 实例的总体目标对齐。
- **局部上下文需求**:需要特别关注的上下文领域(如"预算、地点、人数"),引导注意力分配。
- **可调用工具**:该阶段专有的业务工具(如场地搜索 API),与 agent 全局工具组成并集。
- **局部机制**:该阶段的特殊策略(如超时限制、结果校验循环)。
- **完成条件** → **完成信号 (NodeCompletion)**:节点完成时产出的结构化结果,是 Edge.guard 读取的唯一判断依据。包含:
- **状态 (status)**:节点内部的语义状态(如 ok、failed、cancelled)。
- **结果 (result)**:节点执行的具体产出数据。
- **语义标注 (agentHint)**:agent 可选的语义提示,辅助边进行语义条件判断。

节点内部允许 agent 进行 ReAct,但**不应隐藏跨节点状态迁移规则**。
节点管理的是"每个 agent 实例在此阶段的状态",而非全局单例。
节点对外的唯一接口是 `execute(instance) => NodeCompletion`,不感知边和路由器的存在。

**节点类型**:

- **agent 节点**:内部通过 agent 实例的 LLM 执行能力(llm_execute)驱动 ReAct 循环,完成生成、判断、解释等语义工作。
- **代码节点**:纯代码执行,不调 LLM。用于 TUI 交互、数据准备、结果校验等确定性逻辑。两种节点对外接口相同。

*Avoid: 阶段, 步骤, 工位, 带工具的 prompt 模板*

## 边 (Edge)

边是回路图中的**状态迁移承载者**。它连接源节点和目标节点。边的评估是**按 agent 实例独立进行**的:一个 agent 实例满足迁移条件时,仅该实例被迁移,不影响同一节点内的其他实例。

边不直接执行迁移。它只做一件事:检测迁移条件,条件满足时**发布一个迁移请求 (Migration Request)**。迁移请求包含目标节点、迁移规则和入口准备配置,交由路由器裁决和执行。

**检测方式**:

- 可由 agent 判断(语义条件)
- 可由代码判断(规则条件)
- 或混合判断(由开发者按场景声明)

**迁移请求内容**:/

- **目标节点**:要迁移到的下一节点。
- **迁移规则**:离开当前节点后,保留哪些上下文字段、丢弃哪些、注入哪些新字段。
- **入口准备**:目标节点需要的技能、工具白名单、注意力引导和阶段约束。

边只做状态转换,不承载重业务逻辑。
*Avoid: 普通连线, UI 箭头, 文档箭头, 隐式超级节点*

## 路由器 (Router)

路由器负责响应节点的完成事件。当节点完成时,路由器收集所有满足条件的边发布的迁移请求,按路由规则选择一条边并执行迁移。
当没有边满足条件时,路由器根据配置机制选择:反馈信息给当前节点让其重试,或切入异常处理流程(如诊断图)。

**路由规则**:

- **优先级优先 (priority-first)**:选择优先级最高的边;同级时取第一个。
- **agent 选择 (agent-choice)**:agent 从所有满足条件的边中自主选择。
- **首匹配 (first-match)**:按注册顺序取第一个满足条件的边。
- **全部满足 (all-satisfied)**:所有满足条件的边同时触发(fork)。

路由器是图级别的配置。每个节点出口处挂一个 Router 和一组 Edge。
*Avoid: 调度器, 编排器, 流程引擎, 路由函数*

## 通讯 (Communication)

通讯是一层**高于机制和工具的抽象**——机制和工具是通讯的具体实现方式,上下文是通讯的作用载体。详见 `communication-design.md`。

**三层架构**:

- **Layer 3 — 通讯抽象**:四种通讯模式 (Pub/Sub, Request/Reply, Action, Shared State),参考 ROS2 的通讯原语设计。
- **Layer 2 — 实现方式**:工具 (agent 主动调用,如 `publish`, `subscribe`, `request`, `read_state`) 和机制 (系统自动执行,如消息分发、状态同步、心跳检测)。
- **Layer 1 — 作用载体**:通讯消息作为上下文条目,注入 agent 的通讯上下文层;入站/出站消息分层隔离,由 Edge 的 migrate 控制留存。

**核心约束**:

- agent 是否允许通讯、以什么方式通讯,由**当前节点的 `prepareEntry.tools` 和 `constraints`** 决定。
- 所有通讯必须留下**可追溯的审计记录**。
- 节点可通过 `constraints` 限制发布 topic、订阅范围、请求目标、频率等。

## 组合规则 (Composition Rules)

agent 实例和节点的两套要素**同时组装**到运行中的 agent 身上,构成该阶段的完整执行配置。

- **目标** = 实例总目标 + 节点子目标(聚焦但不覆盖)
- **上下文** = 实例全局记忆 + 节点局部注意力引导 (+ 通讯上下文层)
- **工具** = 实例全局工具 ∪ 节点工具(含通讯工具,如 publish/request/read_state),并可附加节点限制策略
- **机制** = 实例全局机制 + 节点局部策略(含通讯机制,如消息分发/心跳检测)
- **通讯能力** = 节点通过 `prepareEntry.tools` 赋予通讯工具,通过 `constraints` 限定通讯范围(topic 白名单、频率等)
- 节点不应直接修改 agent 实例的总目标或持久身份,全局目标的变更须通过边的触发交由实例的"目标重规划"机制处理。
- 多 agent 下,上述组合对每个实例独立生效;共享环境(Shared State、Message Bus)和通讯设施作为跨实例协作的基础。
