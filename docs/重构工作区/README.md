# 文档重构工作区

本目录不是正式用户文档，而是给多个无上下文 agent 使用的任务说明区。每个子目录的 `README.md` 都是一份可独立执行的工作包；agent 应按其中指定的目标路径修改正式文档，不要把最终文档写回本工作区。

## 重构目标

将当前以实施历史和 Runtime 内部结构为中心的文档，重构为以外部 SDK 开发者为第一读者的文档系统：

1. 新用户能从根 README 在十分钟内运行第一张图。
2. 概念文档先解释业务含义，再介绍英文类型名。
3. 使用指南按“我要完成什么”组织，不按源码文件组织。
4. 完整 API、内部投影协议和维护者信息与入门内容分离。
5. 当前实现、未来研究和历史演进不再混写。
6. 所有能力声明以当前代码和测试为准。

## 全局事实来源

发生冲突时按以下优先级判断：

1. `src/type.ts` 中的公共类型。
2. `src/adapter/loop-graph-extension.ts`、`mechanism-runtime.ts`、`projection.ts` 的真实行为。
3. 测试中的可验证契约。
4. `docs/adr/` 中仍然有效的架构决策。
5. 当前设计和形态文档。
6. 归档文档只能解释历史，不能作为当前能力依据。

## 全局写作规则

- 第一读者是第一次使用 SDK 的外部 TypeScript 开发者。
- 中文正文可以保留英文 API 名，但首次出现必须先用一句中文解释。
- 不把 `scopeId`、`visit`、GraphCallScope、frame baseline 等内部词汇放进入门路径。
- 不宣称并行 Agent、多 Agent 通讯、会话持久恢复已经实现。
- 不复制整段公共类型；参考文档可以摘录最小签名，并链接到源码。
- 示例使用当前开放的 frame 结构，不暗示 `nodeId/status/summary/result` 是必填字段。
- 不按 Phase 编号组织面向用户的内容。
- 每个新增示例都必须通过 TypeScript 类型检查，或明确标注为伪代码。

## 完成定义

全部任务完成后，正式文档应形成四层：README/快速开始、概念、任务指南与 API 参考、维护者内部文档。归档保留历史，但不出现在新用户默认阅读顺序中。
